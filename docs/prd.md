## 1. 프로젝트 개요

### 1.1 프로젝트명
**마이링크 (MyLink)** — 나만의 링크를 한 곳에 모아 공유하는 올인원 링크 허브

### 1.2 프로젝트 목적
SNS 바이오에 하나의 URL만 넣을 수 있는 제약을 해결합니다.  
마이링크는 사용자가 자신의 모든 소셜 미디어, 포트폴리오, 블로그 등 **여러 링크를 한 페이지에 모아서 공유**할 수 있는 링크트리(Linktree) 클론 서비스입니다.

### 1.3 대상 사용자

| 구분 | 설명 |
|------|------|
| **1차 사용자** | 자신만의 온라인 존재감(Presence)을 구축하고 여러 링크를 한 곳에서 공유하고 싶은 사용자 |
| **2차 사용자** | 마이링크 페이지를 방문하여 링크를 클릭하는 일반 방문자 |

### 1.4 기술 스택 (참고)

| 항목 | 기술 |
|------|------|
| Frontend | Next.js (App Router), TypeScript, Tailwind CSS |
| Backend / DB | Firebase (Firestore, Authentication) |
| 인증 | Google 소셜 로그인 (Firebase Auth) |
| 배포 | Vercel 또는 Firebase Hosting |

## 4. 각 기능의 상세 설명

---

### F-01. Google 소셜 로그인

**목적**: 사용자가 별도의 회원가입 절차 없이 Google 계정으로 간편하게 로그인합니다.

**상세 요구사항**:
- Firebase Authentication의 Google 제공자(Provider)를 사용합니다.
- 최초 로그인 시 자동으로 사용자 프로필이 Firestore에 생성됩니다.
- 로그인 상태는 세션을 통해 유지되며, 브라우저를 닫아도 재방문 시 로그인 상태가 유지됩니다.
- 로그아웃 기능을 제공합니다.

**처리 흐름**:
1. 사용자가 "Google로 로그인" 버튼 클릭
2. Google OAuth 팝업 표시
3. 인증 성공 시 → Firestore에 사용자 문서 존재 여부 확인
4. 신규 사용자 → 프로필 문서 자동 생성
   - `displayName`: Gmail 주소의 `@` 앞부분을 추출하여 설정 (예: `yangdogyun0421@gmail.com` → `yangdogyun0421`)
   - `username`: Google 계정의 표시 이름을 가져와 설정 (예: `양도균`)
   - `photoURL`: Google 프로필 사진 URL
5. 기존 사용자 → 대시보드로 리다이렉트

---

### F-02. 프로필 설정

**목적**: 사용자가 공개 페이지에 표시될 개인 정보를 관리합니다.

**설정 항목**:

| 필드 | 타입 | 필수 여부 | 제약 사항 |
|------|------|-----------|-----------|
| displayName (URL 식별자) | 텍스트 | 필수 | 영문 소문자, 숫자, 하이픈만 허용. 최대 30자. 공개 페이지 URL로 사용됨. 중복 불가 |
| username (닉네임) | 텍스트 | 필수 | 최대 30자. 프로필 페이지에 표시되는 실제 이름/닉네임 |
| 소개글 (Bio) | 텍스트 | 선택 | 최대 150자 |
| 프로필 이미지 | URL | 자동 | Google 계정 프로필 사진을 그대로 사용 (별도 업로드 없음) |

**상세 요구사항**:
- displayName, username, 소개글(Bio) 모두 대시보드에서 **인라인 편집**으로 수정합니다. (텍스트를 클릭하면 즉시 편집 모드로 전환)
- **displayName**: 공개 페이지 URL 경로로 사용되는 고유 식별자입니다. (예: `도메인/yangdogyun0421`)
  - 초기값: Gmail 주소의 `@` 앞부분을 자동 추출하여 설정합니다. (예: `yangdogyun0421@gmail.com` → `yangdogyun0421`)
  - 영문 소문자, 숫자, 하이픈(`-`)만 허용합니다.
  - 변경 시 **중복 검사**를 실시간으로 수행합니다.
  - 변경 시 기존 공개 URL이 함께 변경됩니다.
- **username**: 공개 프로필 페이지에서 방문자에게 보이는 이름/닉네임입니다. (예: `양도균`)
  - 초기값: Google 계정의 표시 이름을 가져와 설정합니다.
  - 자유롭게 수정할 수 있습니다.
  - 고유성 제약 없음 (같은 username을 여러 사용자가 사용 가능).
- 프로필 이미지는 Google 계정 사진을 그대로 사용합니다. **별도 이미지 업로드 기능은 제공하지 않습니다.**

---

### F-03. 링크 CRUD

**목적**: 사용자가 공개 페이지에 표시할 링크 목록을 자유롭게 관리합니다.

**상세 요구사항**:
- **추가(Create)**: "링크 추가" 버튼 클릭 시 제목과 URL을 입력하는 폼이 나타납니다.
- **수정(Update)**: 기존 링크의 제목, URL을 **인라인 편집**으로 수정합니다. (텍스트를 클릭하면 즉시 편집 모드로 전환)
- **삭제(Delete)**: 삭제 전 확인 다이얼로그를 표시합니다.
- URL 입력 시 기본적인 유효성 검증을 수행합니다. (http:// 또는 https://로 시작하는지 확인)
- **링크 개수 제한 없음**: 사용자는 원하는 만큼 링크를 추가할 수 있습니다.
- 링크 순서는 등록된 순서(createdAt)대로 표시됩니다.

---

### F-04. 링크 파비콘 아이콘

**목적**: 각 링크에 해당 웹사이트의 파비콘(favicon)을 자동으로 표시하여 시각적 구분을 제공합니다.

**동작 방식**:
- **Google Favicon API**를 사용하여 파비콘을 가져옵니다.
- API URL 형식: `https://www.google.com/s2/favicons?domain={도메인}&sz=64`
- 링크의 URL에서 도메인을 추출한 뒤, 위 API를 통해 파비콘 이미지를 실시간으로 로드합니다.
- 별도로 파비콘 URL을 DB에 저장하지 않고, 클라이언트에서 도메인 기반으로 동적 생성합니다.
- 파비콘을 가져올 수 없는 경우, 기본 링크 아이콘(🔗)을 대체 표시합니다.

**표시 위치**:
- 공개 프로필 페이지의 각 링크 카드 왼쪽에 파비콘 아이콘을 표시합니다.
- 대시보드의 링크 목록에서도 동일하게 파비콘을 표시합니다.

---

### F-05. 공개 프로필 페이지

**목적**: 방문자가 사용자의 링크 목록을 볼 수 있는 공개 페이지입니다.

**페이지 URL 형식**: `도메인/{displayName}` (예: `mylink.com/양도균`)

> [!IMPORTANT]
> URL 경로에 displayName을 직접 사용합니다. 한글 등 비ASCII 문자가 포함될 수 있으므로 URL 인코딩/디코딩 처리가 필요합니다.

**표시 요소**:
1. 프로필 이미지 (원형, Google 계정 사진)
2. username (닉네임 / 표시 이름)
3. 소개글 (Bio)
4. 링크 목록 (등록 순서대로, 각 링크에 파비콘 아이콘 포함)

**상세 요구사항**:
- **비로그인 상태에서 접근 가능** (공개 페이지이므로 인증 불필요)
- 링크 클릭 시 **새 탭**으로 열립니다.
- 존재하지 않는 displayName 접근 시 **404 페이지**를 표시합니다.
- 페이지 로딩 속도를 위해 **SSR(Server-Side Rendering)** 또는 **ISR(Incremental Static Regeneration)** 을 활용합니다.

---

### F-06. 반응형 디자인

**목적**: 어떤 기기에서든 최적의 경험을 제공합니다.

**Breakpoint 기준**:

| 기기 | 너비 | 레이아웃 |
|------|------|----------|
| 모바일 | < 640px | 1단 세로 레이아웃, 터치 친화적 |
| 태블릿 | 640px ~ 1024px | 적당한 여백, 중간 크기 카드 |
| 데스크탑 | > 1024px | 넓은 레이아웃, 사이드바 기반 대시보드 |

**적용 대상**:
- 공개 프로필 페이지 (방문자용)
- 대시보드 / 링크 관리 페이지 (소유자용)
- 로그인 페이지
- 마케팅 랜딩 페이지

## 5. 데이터베이스 모델링 (Firestore)

서브 컬렉션을 활용하여 사용자별 데이터를 구조화합니다.

### 5.1 컬렉션 구조

```
users (컬렉션)
  └─ {userId} (문서)
       ├── displayName: string          // URL 고유 식별자 (예: "dokyun")
       ├── username: string             // 프로필 표시 닉네임 (예: "양도균")
       ├── email: string                // 이메일 (Google)
       ├── photoURL: string             // 프로필 이미지 URL (Google 계정 사진)
       ├── bio: string                  // 소개글
       ├── createdAt: timestamp         // 가입 일시
       ├── updatedAt: timestamp         // 최종 수정 일시
       │
       └── links (서브 컬렉션)
             └─ {linkId} (문서)
                  ├── title: string       // 링크 제목
                  ├── url: string         // 링크 URL
                  └── createdAt: timestamp // 생성 일시
```

### 5.2 displayName 중복 방지

displayName을 URL 경로로 사용하므로 고유성이 보장되어야 합니다.

| 방법 | 설명 |
|------|------|
| **별도 usernames 컬렉션** | `usernames/{displayName}` 문서에 `userId`를 저장하여 닉네임→사용자 매핑 및 중복 방지에 활용 |

```
usernames (컬렉션)
  └─ {displayName} (문서)
       └── userId: string   // 해당 닉네임을 소유한 사용자 ID
```

> [!NOTE]
> displayName 변경 시 기존 `usernames` 문서를 삭제하고 새 문서를 생성하는 트랜잭션 처리가 필요합니다.
